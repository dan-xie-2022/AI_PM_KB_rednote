## 核心要点

- Agent 评估不是单看回答正确率，而是一整套指标体系
- 一个靠谱的 Agent 需同时满足：结果对、过程稳、边界清楚、效率可接受、业务有价值
- 核心评估指标分六个维度：任务完成效果、稳定性、过程质量、安全边界、效率、业务价值
- 有四个常见误判坑：只看 Demo、只看最终结果、只看离线分数、误以为"会说"就是"会做"
- 实用五层评估框架：结果 → 过程 → 稳定性 → 安全 → 效率&业务

## 详细内容

### 一、什么叫"靠谱"？——五个必要条件

一个 Agent 要称得上靠谱，需同时满足以下五点：

| 条件 | 说明 |
|------|------|
| **结果对** | 任务完成且质量过关 |
| **过程稳** | 不是今天能用、明天就用不了，需长期稳定 |
| **边界清楚** | 知道什么时候该停、该追问、该转人工 |
| **效率可接受** | 速度与成本在业务可承受范围内 |
| **业务有价值** | 真正解决业务问题，而非仅技术演示 |

### 二、核心评估指标——六个维度

**1. 任务完成效果**
- 任务完成率：有没有把事情做成
- 目标达成率：做成的程度如何
- 结果质量分：通过人工评审或规则打分

> 完成率看"做没做成"，质量分看"做得好不好"，二者缺一不可。

**2. 稳定性**
- 很多 Agent Demo 表现亮眼，但上线后极不稳定
- 关键不只是"会不会错"，而是"错了之后会不会失控"

**3. 过程质量**
Agent 与普通问答模型最大的区别在于"有过程"——拆解任务、调用工具、理解上下文、做决策。因此需关注：
- 规划合理率
- 工具调用成功率
- 工具选择准确率
- 是否存在大量无效动作（如反复查询无用信息）

> 表面做成了，过程却脆弱或成本高，规模化应用时必然出大问题。

**4. 安全边界**
- 幻觉率（Hallucination rate）
- 澄清率（需要追问的比例）
- 拒绝回答率（对超出能力边界的请求是否能正确拒绝）

**5. 效率指标**
- 单次成功成本 = 总成本 ÷ 成功完成的任务数
- 太慢、太贵，即使准确率高也不值得部署

**6. 业务价值**
- 回归业务本身看业务指标
- 核心问题：有没有帮助业务"多赚、少花、更快、更稳"

### 三、四个常见误判坑

1. **只看 Demo，不看真实任务级别** — Demo 往往是精心设计的最优场景
2. **只看最终结果，不看过程** — 过程脆弱的 Agent 无法规模化
3. **只看离线分数，不看线上表现** — 测试集与真实环境存在分布差异
4. **误以为"会说"变成了"会做"** — Agent 解释得漂亮 ≠ 结果能落地

### 四、实用五层评估框架

```
第一层：结果        ← 做成了没有？质量如何？
第二层：过程        ← 路径合理吗？工具用对了吗？
第三层：稳定性      ← 能稳定复现吗？异常时会失控吗？
第四层：安全        ← 幻觉率、边界识别是否合格？
第五层：效率 & 业务  ← 成本可接受吗？业务价值是否兑现？
```

## 金句摘录

> "大部分人第一反应都是看回答准不准，但一旦这么打，面试官基本就能判断你大概率没有真正落地做过 Agent。"

> "偶尔成功那不叫靠谱；成功了但成本太高，也不叫靠谱；平时很好，一到异常就乱来，也不叫靠谱。"

> "真正要看的不是它说得有多漂亮，而是要看它最后的结果能不能落现，出错时候能不能恢复。"
