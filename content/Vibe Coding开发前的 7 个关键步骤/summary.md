## 内容概述

> 作者亲身经历了 Vibe Coding 项目从混乱崩溃到顺利完成的对比，总结出7个在让AI写第一行代码之前必须完成的准备步骤。核心观点：Vibe Coding 失败不是 AI 的问题，而是跳过了准备阶段——就像没有图纸就开工盖房，盖到一半只能推倒重来。

## 核心要点

- 在 AI 写第一行代码之前，先花 30 分钟到 1 小时做准备，可以避免后续几小时甚至几天的返工
- PRD 是你和 AI 之间的"合同"，改 PRD 花 5 分钟，改代码可能花 5 个小时
- 项目目标（自用 vs 产品）直接决定技术方案，必须提前明确
- 技术栈原则：选主流的、选成熟的、选合适的——不要杀鸡用牛刀
- 架构文档保存在项目根目录，Cursor / Claude Code 等工具会自动读取，之后每次对话 AI 都"知道"项目规范
- Git 是 AI 编程的"后悔药"，每完成一个功能就 commit 一次，出错可以精确回滚
- 开发时先做 MVP，小步迭代，每次只做一个功能

## 逐图内容

### 第1张
**封面**
标题：Vibe Coding 开发前的 7 个关键步骤（全文4377字，阅读需9分钟）
作者自述：纯 Vibe Coding 一个项目后发现 AI 写的代码越改越乱、项目崩了；清空代码从头再来，这次顺利完成。引出：问题不在 AI，在于跳过了准备环节。

### 第2张
**核心比喻 + 第一步开篇**
"就像盖房子，没有图纸就开工，盖到一半发现门窗位置不对，只能推倒重来。"
建议打开 Claude 或 ChatGPT，按 7 个步骤逐步准备。
**第一步：把想法聊清楚** — 完整地（不是简洁地）把想法说出来，例如做记账工具要说清楚：为什么做、核心功能、目标用户、使用场景。

### 第3张
**第一步（续）：想法模板**
四个关键问题：为什么做？核心功能？谁会用？使用场景？
这个过程帮你自己理清思路——很多时候以为想清楚了，说出来才发现到处是模糊地带。
实用模板：
```
| 我想做一个 [产品名称]
| 主要是为了解决 [什么问题]
| 目标用户是 [谁]
| 核心功能包括：[功能一][功能二][功能三]
| 我希望它能 [达到什么效果]
```

### 第4张
**第二步：要一份 PRD**
聊完想法后，让 AI 整理成产品需求文档（PRD）。说："帮我把刚才聊的内容整理成一份产品需求文档。"
PRD 通常包括：产品定位、核心功能列表、用户流程、技术要求。

### 第5张
**第二步（续）：认真读 PRD**
逐条对照原始想法，确认 AI 理解的和你想的是否一致。不对的地方现在就改。
把 PRD 当作你和 AI 的"合同"。
**第三步：明确项目目标** — 自用、给朋友用、还是做成正式产品？这个问题直接决定技术方案。

### 第6张
**第三步（续）：目标对比**
自用：SQLite 即可、不需要登录系统、不用考虑性能优化、本地运行。
做成产品：需要 PostgreSQL、用户注册登录、数据安全隐私、云服务器部署、多人并发性能。
结论：开发前一定要告诉 AI 你的真实目标，不要一开始搞太复杂。

### 第7张
**第三步（续）：目标表述模板**
```
这个项目主要是我自己用，目标是低成本实现，在本地运行就可以，不需要考虑多人使用的情况。
```
或：
```
这个项目我想做成可以 release 的产品，需要考虑用户数据安全、性能优化和未来扩展性。
```

### 第8张
**第四步：确定技术栈**
目标明确后，让 AI 推荐技术栈。实用 prompt 模板：
```
| 我想做一个 [项目描述]
| 项目目标是 [自用/商用/学习]
| 我的技术水平是 [完全零基础/有一些基础/比较熟练]
| 请推荐合适的技术栈，并说明为什么这样选择。
```
AI 会给出前端框架、后端语言、数据库、工具库建议。

### 第9张
**第四步（续）：选技术栈原则 + 第五步开篇**
选技术栈三原则：选主流的（问题容易找到答案）、选成熟的（工具链完善坑少）、选合适的（匹配项目规模，不要过度工程化）。
例：简单网页工具用 React + Node.js + SQLite 就够了，不需要 Kubernetes、微服务。
**第五步：要一份架构设计** — 让 AI 设计项目目录结构、核心模块划分、数据模型设计、代码规范建议。

### 第10张
**第五步（续）：架构设计输出示例**
AI 会输出标准目录结构：
```
project/
  src/
    components/   # UI 组件
    services/     # 业务逻辑
    utils/        # 工具函数
    config/       # 配置文件
  public/
  tests/
```

### 第11张
**第五步（续）：模块划分 + 数据模型 + 代码规范**
模块划分示例（记账工具）：数据录入模块、数据展示模块、统计分析模块、数据导出模块。
数据模型：定义存储哪些数据及其关系。
代码规范：组件命名 PascalCase、函数命名 camelCase。

### 第12张
**第五步（续）：架构哲学**
文件命名 kebab-case，每个函数前写注释。
不必追求"完美架构"——好的架构是"刚刚好"的：满足当前需求，有一定扩展空间，但不为可能永远用不到的功能增加复杂度。架构可以随项目发展逐步调整。
**第六步：整理项目文档** — 把 PRD、技术栈、架构设计整理成文档保存到项目里。

### 第13张
**第六步（续）：建议创建的文档**
- `README.md`：项目简介、如何运行、技术栈
- `PRD.md`：产品定位、核心功能列表、用户使用流程
- `ARCHITECTURE.md`：技术栈选择及原因、项目目录结构

### 第14张
**第六步（续）：文档的价值 + 第七步开篇**
ARCHITECTURE.md 还包含：核心模块说明、数据模型设计、代码规范。
关键价值：使用 Cursor / Claude Code 等工具时，AI 会自动读取项目根目录文档，之后每次对话都已经"知道"项目架构和规范，无需反复上传。
重要提醒：文档要跟着项目更新，否则 AI 会按过时文档理解项目，导致新代码和现有代码对不上。
**第七步：设置 Git 版本管理**

### 第15张
**第七步：Git 版本管理详解**
Git = 给项目创建"存档点"，就像游戏存档系统。
为什么 AI 编程特别需要 Git：AI 修改代码时很容易在解决一个 bug 的同时破坏其他功能（代码有复杂依赖关系）。有了 Git，每完成一个功能就创建存档点，出问题随时回到安全状态。

### 第16张
**第七步（续）：Git 基本用法**
```
# 项目开始时
git init
git add .
git commit -m "项目初始化"

# 每完成一个功能
git add .
git commit -m "完成了 XX 功能"
```
commit 信息要写清楚这次改了什么，方便以后查找。

### 第17张
**第七步（续）：回滚 + 实用节奏**
```
git log                              # 查看历史记录
git reset --hard [某个commit的ID]   # 回到那个版本
```
实用节奏：让 AI 列出开发计划，按计划一步步开发，每完成一个独立功能就 commit 一次。
示例：
- ✅ 完成数据录入功能 → git commit
- ✅ 完成数据展示功能 → git commit
- ✅ 完成统计功能 → git commit
- ❌ 添加导出功能后项目崩了 → git reset 回到添加前

### 第18张
**完整流程总结**
七个步骤串联：
01. 跟 AI 聊想法：把脑子里的想法完整说出来
02. 要 PRD 并确认：确保 AI 理解的跟你想的一致
03. 明确项目目标：是自用还是要做成产品
04. 确定技术栈：让 AI 根据目标推荐合适的技术
05. 要架构设计：获取目录结构、模块划分、数据模型
06. 整理项目文档：把这些信息保存成文档
07. 设置 Git：git init，准备好版本管理

### 第19张
**从准备到开发：4个原则**
这个准备过程需要 30 分钟到 1 小时，但能帮你避免后续几小时甚至几天的返工。
1. **先做 MVP（最小可用版本）**：先做能跑、能用、实现核心功能的版本，后面再优化。
2. **小步迭代**：每次只做一个功能，做完测试没问题再做下一个。

### 第20张
**从准备到开发（续）**
3. **勤 commit**：每完成一个小功能就 commit 一次，这是安全网，别省。
4. **验证 AI 的代码**：要自己测试，不要盲目相信，也不要过度怀疑，测试是最好的验证方式。
对完全零基础新手的建议：从简单项目开始（待办事项、简易记账本、个人博客），目的是把整个流程走一遍，体会每个步骤的作用。

### 第21张
**结尾**
做完一个项目，流程就刻在脑子里了，也有了 Vibe Coding 的感觉。
别害怕出错——有 Git 在，出错了就回滚，没什么大不了的。多试几次，经验就来了。
