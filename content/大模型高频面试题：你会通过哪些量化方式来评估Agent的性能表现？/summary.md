## 内容概述

> 本视频系统讲解了评估 Agent 性能的五大量化维度，以及工程实践中三个最难解决的痛点与对应专家级解决方案。面向大模型高级面试场景，直接给出"让面试官刮目相看"的完整答题框架。

## 核心要点

- 评估 Agent 不能只看准确率/ROUGE，需要覆盖「感知-规划-行动」完整闭环
- 五大维度：任务完成度、推理规划能力、工具调用能力、工程效率、安全鲁棒性
- 三大工程难点：开放性任务自动打分、多轮误差累积、动态环境不可控
- 最容易被忽略但最致命的指标：**误差累积**和**幻觉操作**
- 面试高分答法：先讲架构逻辑，再点出易忽略的关键指标，最后说出自动化评测体系

## 详细内容

### 一、五大评估维度

**维度 1：任务完成度（端到端结果）**
- **任务成功率**：给定 N 个任务，Agent 在限定步骤内完全完成几个
- **目标达成度**：任务未全部完成时，根据子任务完成情况给 0~1 的连续分，避免一刀切打零分

**维度 2：推理与规划能力**
- **平均规划步数**：同任务 A 模型 3 步完成、B 模型绕 10 步，规划能力高下立判
- **自我纠错率**：遇到报错能否反思后换策略继续，直接决定 Agent 的韧性
- **无效动作占比**：输出格式错误、调用不存在的工具，均计入无效动作

**维度 3：工具调用能力**
- **工具选择准确率**：面对海量 API 能否精准命中正确工具
- **参数提取准确率**：必填项是否写对、数据类型是否正确
- **API 错误率**：实际执行中触发系统报错的频率

**维度 4：工程效率**
- **任务延迟 / 首动作耗时（T-TFA）**：用户等半分钟才看到 Agent 第一个动作，体验极差
- **Token 消耗量**：完成单次任务平均花多少 Token，直接影响 ROI

**维度 5：安全与鲁棒性**
- **幻觉动作率**：有没有调用从未存在的工具、尝试未授权的操作
- **危险指令拦截率**：面对恶意注入攻击，Agent 能否守住底线

### 二、三大工程难点与解决方案

**难点 1：开放性任务如何自动打分**

问题：「帮我策划旅行并预订」没有唯一标准答案，硬比对无效。

解决方案：
- 引入 **LLM as Judge**：用更强的模型从逻辑性、可行性等维度评分
- 大任务拆检查点：每过一道关卡给部分分，不用等最终结果

**难点 2：多轮交互误差累积效应**

问题：第一步数据查错 → 后续步步接错 → 最终得零分，但不代表规划能力差。

解决方案：**Teacher Forcing** —— 框架强行把状态重置，将正确的中间结果喂给模型，再独立评估下一步，从而隔离每步的规划能力。

**难点 3：动态环境导致评估不可靠**

问题：今天测试的网页明天改版，API 下线，昨天的测试结果今天就不作数。

解决方案：**静态容器沙盒** —— 把 Agent 关进独立隔离容器，或对 API 响应进行 Mock，保证每次评估输入的绝对等价性。

### 三、面试答题框架（高分模板）

> "Agent 本质上是感知-规划-行动的闭环系统，因此评估不能局限于传统文本比对，必须兼顾最终任务达成与中间思维轨迹。在实际工程中，最容易被忽略但最致命的是误差累积和幻觉操作。我会优先建立包含**自我纠错率**和**参数提取准确率**的自动化测试监控体系，同时采用轻量级 proxy 数据集进行预筛，平衡评估成本与迭代效率。"
